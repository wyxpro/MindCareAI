# 前端组件与页面文档撰写与审查

## 核心原则：文档即代码（Docs as Code）

前端文档遵循"文档即代码"理念：

- **文档重在指引** - 说明组件职责、使用方式、数据来源，而非复制代码实现
- **保持文档简洁** - 通过代码位置引用让开发者直接查看实现
- **AI 编写质量更高** - 严格禁止使用 TypeDoc、JSDoc CLI 等自动生成工具

## 功能

### 1. 组件文档撰写

为 Vue/React 组件生成符合规范的文档。

**触发示例：**
```
"帮我为 Button 组件编写文档"
"根据 src/components/UserCard.vue 生成文档"
"我需要为业务组件写文档"
```

**执行流程：**

1. **读取组件代码** - 分析组件：
   - 组件位置和类型（公共组件/业务组件）
   - Props/Events/Slots 定义
   - 数据来源（API 调用、状态管理）
   - 权限控制

2. **检查代码注释完整性** - 评估是否有足够信息生成文档：
   - ✅ **注释完整**：代码中有 JSDoc/TypeScript 接口注释，组件职责清晰
   - ⚠️ **注释不全**：缺少关键注释，组件用途或 Props 说明模糊
   - ❌ **注释缺失**：几乎没有注释，无法理解组件功能

3. **信息确认（如果注释不全）** - 使用问答+选项形式引导用户确认：
   - 组件的具体用途和职责
   - 数据来源（调用哪个 API 接口？使用哪个状态管理？）
   - 是否有权限控制要求
   - 是否有特殊验证逻辑

   ⚠️ **严格禁止盲目生成**：如果代码注释不足且用户未确认信息，不得直接生成文档

4. **生成文档内容** - 更新 `docs/components/README.md`，包含：

   **必须包含的内容：**
   - **组件位置** - 文件路径
   - **功能说明** - 用 1 句话概括组件用途
   - **使用示例** - 引用组件内 JSDoc 注释
   - **Props/Events/Slots 文档** - 说明通过代码注释查看详情

   **严格禁止的内容：**
   - ❌ 复制粘贴具体代码实现
   - ❌ 详细的 Props 列表（应在代码注释中维护）
   - ❌ 使用 TypeDoc 等自动生成工具

5. **文档模板示例：**

```markdown
# 组件总览

## 公共组件

### Button（按钮组件）
- **组件位置：** `src/components/common/Button.vue`
- **功能说明：** 通用按钮，支持多种样式和尺寸
- **使用示例：** 见组件内 JSDoc 注释
- **Props 文档：** 见组件源码中的 TypeScript 接口定义

### Input（输入框组件）
- **组件位置：** `src/components/common/Input.vue`
- **功能说明：** 通用输入框，支持验证和错误提示
- **验证逻辑：** 集成 `src/utils/validator.ts`

## 业务组件

### UserCard（用户卡片）
- **组件位置：** `src/components/business/UserCard.vue`
- **功能说明：** 展示用户信息卡片
- **数据来源：** 调用 `/api/user/profile` 接口
- **权限控制：** 需登录后访问

详细 Props、Events、Slots 说明见组件源码中的 JSDoc 注释。
```

### 2. 页面文档撰写

为前端页面生成功能概述文档。

**触发示例：**
```
"帮我为用户管理页面编写文档"
"根据 src/pages/Dashboard.vue 生成文档"
```

**执行流程：**

1. **读取页面代码** - 分析页面：
   - 页面路由路径
   - 主要功能模块
   - 使用的组件
   - API 接口调用

2. **检查代码注释完整性** - 评估是否有足够信息生成文档：
   - ✅ **注释完整**：代码中有清晰的注释说明页面功能和业务逻辑
   - ⚠️ **注释不全**：缺少关键功能说明，业务流程模糊
   - ❌ **注释缺失**：几乎没有注释，无法理解页面用途

3. **信息确认（如果注释不全）** - 使用问答+选项形式引导用户确认：
   - 页面的主要功能有哪些
   - 调用了哪些 API 接口
   - 使用了哪些组件
   - 是否有特殊的业务逻辑

   ⚠️ **严格禁止盲目生成**：如果代码注释不足且用户未确认信息，不得直接生成文档

4. **生成文档内容** - 更新 `docs/pages/README.md`，包含：

```markdown
# 页面列表和功能概述

## 用户管理

### 用户列表页（/users）
- **页面位置：** `src/pages/user/UserList.vue`
- **主要功能：** 用户查询、新增、编辑、删除
- **使用组件：** UserTable、UserForm、SearchBar
- **接口调用：** `/api/user/list`、`/api/user/create`

### 用户详情页（/users/:id）
- **页面位置：** `src/pages/user/UserDetail.vue`
- **主要功能：** 查看用户详细信息、编辑资料
- **使用组件：** UserCard、UserForm
- **接口调用：** `/api/user/profile`、`/api/user/update`
```

### 3. 文档审查

审查现有前端文档是否符合团队规范。

**触发示例：**
```
"审查 docs/components/README.md"
"检查我写的组件文档"
"这个文档有什么问题？"
```

**审查清单：**

✅ **应该有的内容：**
- [ ] 组件/页面位置（文件路径）
- [ ] 功能说明（1 句话概括）
- [ ] 使用示例引用（指向代码注释）
- [ ] 数据来源说明（API 接口/状态管理）
- [ ] 特殊说明（权限控制、验证逻辑等）

❌ **不应该有的内容：**
- [ ] 复制粘贴的代码实现
- [ ] 详细的 Props/Events 列表（应在代码中维护）
- [ ] TypeDoc 自动生成的文档

⚠️ **文档质量标准：**
- [ ] 重在说明"做什么"（功能职责）和"在哪里"（代码位置）
- [ ] 不详细说明"怎么做"（具体实现留给代码和代码注释）
- [ ] 通过代码位置引用指引开发者
- [ ] 保持简洁，避免冗长描述

**审查输出格式：**

```markdown
## 审查结果

### ✅ 符合规范的部分
- 组件位置标注清晰
- 功能说明简洁明了

### ❌ 需要改进的部分
- [问题1] 包含了详细的 Props 列表（第 XX 行）
  - 建议：删除列表，改为"见组件源码中的 TypeScript 接口定义"
- [问题2] 使用了 TypeDoc 自动生成的内容
  - 建议：删除自动生成内容，手动编写简洁文档

### 📝 改进建议
[具体的修改建议]
```

## 使用场景

**场景 1：新组件开发完成**
```
开发者："我刚完成 UserCard 组件开发，帮我生成文档"
→ 读取 src/components/business/UserCard.vue 代码
→ 更新 docs/components/README.md
```

**场景 2：页面文档编写**
```
开发者："为 Dashboard 页面编写文档"
→ 读取 src/pages/Dashboard.vue 代码
→ 更新 docs/pages/README.md
```

**场景 3：文档审查**
```
开发者："审查 docs/components/README.md"
→ 读取文档内容
→ 对照审查清单
→ 输出审查结果和改进建议
```

## 重要提示

1. **文档位置固定**
   - 组件文档：`docs/components/README.md`
   - 页面文档：`docs/pages/README.md`

2. **严格禁止自动生成工具**
   - ❌ 禁止使用 TypeDoc、JSDoc CLI 等工具
   - ✅ 由 AI（Claude Code）配合开发者手动编写

3. **代码注释是核心**
   - Props/Events/Slots 详情应写在代码的 JSDoc 注释中
   - 文档只需引用"见组件源码中的 JSDoc 注释"

4. **与代码同步**
   - 组件变更时必须同步更新文档
   - 重点更新功能说明和数据来源，不更新代码细节

5. **相关 Skills 协作**
   - 提交前必须使用 `react-code-review` 或 `vue-code-review` 进行代码审查
